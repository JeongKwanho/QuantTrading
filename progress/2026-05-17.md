## 1. System Design

### Decisions Made

| Item | Decision |
|------|----------|
| Language | Python 3.x |
| Exchange | Binance Futures (USDT-M) only — extensible via BaseBroker |
| API Framework | FastAPI + uvicorn |
| Database | PostgreSQL (persistent) + Redis (cache) — deferred, using in-memory mock for now |
| Frontend | React 18 + TypeScript (not started yet) |
| Authentication | API Key via `X-API-Key` header |
| Execution | uvicorn standalone |

### Architecture Principles

- Strategies return `Signal` objects only — no direct broker access
- All broker implementations share `BaseBroker` interface — swap exchange by changing `.env`
- Frontend communicates only through `backend/api/` via REST + WebSocket
- `backtest/` uses `broker/mock.py` — never the live broker
- Folder isolation: each layer is worked on independently

### Key Files Explained

**`backend/broker/base.py`**
- `Signal` — strategy output: direction (BUY/SELL), quantity, price (None = market order)
- `BaseBroker` — abstract class with `place_order`, `cancel_order`, `get_order`, `get_open_orders`, `get_balances`, `get_positions`, `get_ticker`, `get_orderbook`, `get_ohlcv`
- Callback hooks: `set_on_fill()`, `set_on_ticker()` — engine registers these to receive real-time events

**`backend/broker/binance.py`**
- Implements all `BaseBroker` methods using `python-binance` `AsyncClient`
- `connect()` creates `AsyncClient` + `BinanceSocketManager`
- `subscribe_ticker()` / `subscribe_user_stream()` — WebSocket streams for price and fill events
- `_parse_order()` — converts raw Binance response to `Order` dataclass

**`backend/api/state.py`**
- `AppState` holds the broker instance
- Falls back to mock data when broker is not connected (no `.env` set)
- Designed for easy swap: replace mock methods with real engine calls later

**`backend/api/main.py`**
- `lifespan` context manager: connects broker on startup, disconnects on shutdown
- Runs with mock data if `BINANCE_API_KEY` is not set
- Swagger UI available at `/docs`
