## 1. FVG Detection Rebuilt

Updated:

```text
patterns/fair_value_gap.py
tools/fvg_chart.py
```

Detection is now simple and observation-oriented:

- Bullish FVG:
  - prior candles were falling,
  - candle 1 high is below candle 3 low,
  - middle candle is larger than both side candles.
- Bearish FVG:
  - prior candles were rising,
  - candle 1 low is above candle 3 high,
  - middle candle is larger than both side candles.

Removed the previous strict structure filter:

- no downtrend-line break requirement,
- no liquidity-low preservation requirement,
- no pivot-based FVG structure requirement.

Added filters:

```text
middle_range_multiplier = 1.2
min_gap_size = 0.0
min_gap_pct = timeframe-specific
```

Timeframe-specific relative gap filter:

```python
MIN_GAP_PCT_BY_TF = {
    "5m": 0.0010,
    "15m": 0.0012,
    "1h": 0.0015,
    "4h": 0.0020,
    "1d": 0.0030,
}
```

Reason:

```text
Absolute gap size does not generalize across BTC, ETH, DOGE, etc.
Relative gap percentage does.
```

## 3. FVGChannelV1 Strategy

Added:

```text
strategies/fvg_channel_v1.py
```

Long setup:

- Entry 1: FVG upper bound, 25%.
- Entry 2: FVG midpoint, 25%.
- Entry 3: FVG lower bound, 50%.
- SL1: lowest low from FVG lookback scope plus the FVG candles.
- TP1: highest high from FVG confirmation until first entry.
- TP2: previous liquidity high above TP1.
- SL2: after TP1, exit remaining position at average entry price.

Short setup:

- Entry 1: FVG lower bound, 25%.
- Entry 2: FVG midpoint, 25%.
- Entry 3: FVG upper bound, 50%.
- SL1: highest high from FVG lookback scope plus the FVG candles.
- TP1: lowest low from FVG confirmation until first entry.
- TP2: previous liquidity low below TP1.
- SL2: after TP1, exit remaining position at average entry price.

Important behavior change:

```text
Once a FVG setup exists, the strategy ignores new FVGs until the current setup
is completed, stopped, or expired.
```

Sizing correction:

```text
Entry sizing now uses the total equity at first entry as the fixed base.
The three entries therefore target 25% / 25% / 50% of the same base equity,
instead of using shrinking free balance after each partial entry.
```

## 11. Live Detection Monitor

Added:

```text
run_live_detection.py
```

Purpose:

```text
Watch selected symbols and timeframes live, store closed candles and detection
snapshots in SQLite, and record when the rolling 15-candle combined detection
changes from 0 to 1.
```

Symbols:

```text
XRPUSDT, BTCUSDT, ETHUSDT, HBARUSDT, RIFUSDT, WLDUSDC
```

Timeframes:

```text
5m, 15m, 1h, 4h, 1d
```

Database:

```text
data/live_detection.sqlite3
```

Tables:

- `candles`
- `detection_snapshots`
- `detection_events`

Behavior:

- Warm up each symbol/timeframe with recent futures candles.
- Use one `CombinedDetectionJudge` per symbol/timeframe.
- Store every closed candle.
- Store every detection snapshot.
- Insert a `detection_events` row only when the state changes from 0 to 1.

- On reconnect, judges are recreated and warmed up again with recent candles.
